# AI Planning Notes

## 결론

Plantgram2의 첫 AI 기능은 별도 AI 메뉴가 아니라 기존 기록 작성 흐름 안에 들어가는 것이 가장 적합하다. 현재 앱의 핵심 가치는 "식물을 잘 관리하게 돕는 것"이므로, AI도 상담 서비스가 아니라 기록 입력 비용을 낮추는 도구로 시작해야 한다.

## 1차 AI 기능

### 자연어 돌봄 기록 입력

사용자가 자연어로 한 문장을 입력하면 AI가 기록 초안을 만든다.

예시 입력:

> 오늘 몬스테라 물 줬고 잎이 조금 처진 것 같아

예시 초안:

| 필드 | 값 |
| --- | --- |
| 식물 | 몬스테라 |
| 기록 유형 | 물주기, 상태 |
| 날짜 | 오늘 |
| 메모 | 잎이 조금 처짐 |
| 저장 방식 | 사용자 확인 후 저장 |

## 넣을 위치

| 위치 | 권장도 | 이유 |
| --- | --- | --- |
| 기록 탭 상단 | 높음 | 전체 기록을 남기는 화면이라 AI 입력 목적이 명확함 |
| 내 식물 상세 기록 액션시트 | 높음 | 식물 선택이 이미 되어 있어 파싱 난이도가 낮음 |
| 내 식물 홈 오늘 할 일 카드 | 중간 | 빠른 완료 처리와 연결 가능 |
| 사전 상세 | 낮음 | 검색/추천 보조는 가능하지만 1차 목적과는 다소 떨어짐 |
| 별도 AI 탭 | 낮음 | 앱 구조가 무거워지고 MVP 초점이 흐려짐 |

## UX 원칙

- AI 결과를 자동 저장하지 않는다.
- 항상 사용자가 확인하고 저장한다.
- 식물명 매칭은 먼저 로컬 내 식물 목록에서 시도한다.
- 확신이 낮으면 "선택 필요" 상태로 보여준다.
- 결과 초안은 기존 기록 저장 UI와 같은 컴포넌트를 재사용한다.
- 실패 시에도 수동 기록 작성으로 바로 이어져야 한다.

## 기술 구조 제안

| 계층 | 역할 |
| --- | --- |
| View | 자연어 입력, 초안 확인 UI |
| Controller | 입력 상태, 파싱 요청, 초안 수정 상태 관리 |
| Repository | Cloud Function 호출과 결과 모델 변환 |
| Firebase Functions | AI API 호출, 프롬프트/스키마 관리, 비용 제한 |
| Firestore | 최종 확정된 기록만 저장 |

앱에 AI API 키를 직접 넣으면 안 된다. Cloud Functions를 통해 호출하고, 사용자별 요청 제한과 로그 정책을 둬야 한다.

## 데이터 스키마 초안

```json
{
  "plantId": "myplant_id_or_null",
  "plantName": "몬스테라",
  "actionTypes": ["water", "condition"],
  "recordedAt": "2026-05-30T09:00:00+09:00",
  "memo": "잎이 조금 처짐",
  "confidence": 0.82,
  "needsUserSelection": false
}
```

## 단계별 적용

| 단계 | 범위 | 목표 |
| --- | --- | --- |
| AI-0 | Storage/기록 경로 정리 | 기존 기록 저장 구조 안정화 |
| AI-1 | 내 식물 상세 AI 입력 | 식물 선택이 쉬운 위치에서 검증 |
| AI-2 | 기록 탭 AI 입력 | 전체 기록 입력으로 확장 |
| AI-3 | Cloud Function 파서 | 비용/보안/스키마 통제 |
| AI-4 | 주간 요약 | 기록 데이터 기반 요약 제공 |

## 하지 않는 것

- AI가 기록을 자동 저장하지 않는다.
- 이미지 질병 진단은 MVP에서 제외한다.
- 범용 식물 상담 챗봇을 첫 기능으로 만들지 않는다.
- 사전 데이터와 충돌하는 관리 조언을 확정적으로 말하지 않는다.
- 사용자의 민감한 원문 로그를 장기 보관하지 않는다.
